Perguntas sobre Git

Perguntas & Respostas

  1. O que são operações de “Commit”, “Push” e “Pop” efetuadas num repositório?

    Commit – é uma gravação de um conjunto de alterações no repositório local. Funciona como um ponto ou “fotografia” do estado dos ficheiros naquele momento.

    O commit contém uma mensagem de commit, o autor, a data e as diferenças (diff) em relação ao estado anterior.

    Push – é a operação que envia os commits do repositório local para o repositório remoto. Serve para partilhar o trabalho com outras pessoas e para manter o repositório remoto sincronizado com o teu.

    Pop (normalmente git stash pop) – está ligado ao comando git stash, que guarda temporariamente alterações que ainda não foram commitadas.

    • git stash: esconde/guarda as alterações num “stash” (pilha), deixando a working directory limpa.
    • git stash pop: traz de volta (reaplica) as alterações guardadas no stash para a working directory e remove esse item da pilha de stashes.

    É especialmente útil quando é preciso mudar de branch rapidamente sem perder aquilo que se estava a fazer.

  2. Quais são as vantagens de usar Branches num repositório Git?

    As branches permitem organizar melhor o desenvolvimento e trazer várias vantagens:

    • Isolamento de trabalho – cada correção de bug, nova funcionalidade ou experimento pode ser desenvolvido numa branch própria, sem mexer no código estável (por exemplo, na branch main).
    • Desenvolvimento paralelo – várias pessoas podem trabalhar em tarefas diferentes ao mesmo tempo, reduzindo conflitos e bloqueios.
    • Maior estabilidade na branch principal – a branch principal mantém-se mais estável e protegida, recebendo apenas código revisto e testado.
    • Experimentação segura – é possível criar branches para testar ideias novas; se não forem úteis, basta apagar a branch sem estragar o código principal.

  3. Nos merges de conteúdos resultantes de um “Pop” podem surgir conflitos. Em que consistem e como podem ser resolvidos esses conflitos?

    Os conflitos consistem em situações em que duas versões do mesmo trecho de código ou texto entram em choque e o Git não consegue decidir automaticamente qual versão deve manter.

    Isto acontece, por exemplo, quando o conteúdo de uma determinada linha foi alterado tanto na branch atual como nas alterações guardadas em stash.

    Para resolver esses conflitos, é necessário:

    • Analisar cuidadosamente as duas versões do código/texto em conflito;
    • Decidir se se mantém a versão atual, a versão do stash, ou se se combina as duas manualmente;
    • Editar o ficheiro, removendo as marcas de conflito do Git e deixando apenas a versão final pretendida;
    • Guardar o ficheiro e marcar o conflito como resolvido (por exemplo, com git add seguido de um commit, conforme o caso).

  4. “Clone” e “Fork” são formas diferentes de replicar um repositório Git. Quais as diferenças entre eles?

    Clone – cria uma cópia local de um repositório remoto existente. Normalmente é usado para obter o código para a tua máquina e trabalhar diretamente sobre esse repositório, fazendo depois push para o mesmo remoto (se tiveres permissões).

    Fork – cria uma cópia do repositório remoto na tua própria conta (por exemplo, no GitHub ou GitLab). Ou seja, é criado um novo repositório remoto, independente do original, mas com o mesmo histórico inicial.

    Em resumo: o clone copia o repositório para o teu computador; o fork copia o repositório para a tua conta, criando um repositório remoto teu, a partir do qual depois também podes fazer clone.
